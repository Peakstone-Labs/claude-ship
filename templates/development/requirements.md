# {FeatureName} - Requirements

## Goal
用户真正想解决的问题 / 想达成的产出。要具体到"触发动作 → 预期结果"，避免抽象概念。

## Non-Goals (Constraints)
- 明确**不做** / 不在范围内的事情
- 硬性边界（性能 / 兼容性 / 合规 / 依赖禁令等）
- 容易被误认为在 scope 内但实际不做的邻近功能

## Success Criteria
按"可验证"标准列出，每条必须能用具体场景 / 数字 / 输出样本来验证：

- 条件 1：...（如"P95 接口延迟 <200ms"）
- 条件 2：...（如"现有集成测试全绿，无新增 5xx"）
- ...

## Context & Assumptions
- 已探索的相关代码区域及要点
- 用户澄清中明确的环境 / 技术假设
- 已知的上游依赖或外部系统状态

## Open Questions (Deferred)
不影响当前 scope 决策、但架构阶段可能要再讨论的问题。每条：
- 问题是什么
- 为什么不在当前 scope 解决
- 架构阶段需要谁决策
